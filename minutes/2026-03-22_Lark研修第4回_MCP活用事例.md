| 項目 | 説明 |
|------|------|
| **MCP（Model Context Protocol）とは** | LLMアプリ（ChatGPT・Claude・Cursorなど）と外部システムをつなぐための**標準通信規格**。HTTPに類似した通信プロトコル。 |
| **MCPがない場合** | 各AIアプリが外部サービス1つひとつに個別接続設定が必要。 |
| **MCPがある場合** | MCPサーバーを1つ挟むことで、ChatGPT・Claude・Cursor等がすべて同一設定で接続可能になる。 |
| **構成要素（5要素）** | ①リモートサービス、②ローカルデータソース、③MCPサーバー、④MCPクライアント、⑤MCPホスト |

> **補足：** 5つの構成要素の詳細は「概念として掴む程度で問題なし」と遠藤氏よりコメントあり。

### 3-2. 活用事例紹介 ― 公式LINEとLark Bitableの連携

**08:20頃 ～ 23:51頃**

遠藤氏より、実際に構築した公式LINEとLark Bitable（ベース）の連携システムについてデモを交えて説明が行われた。

#### システム構成

```
公式LINE
  ↓ Webフック（友達追加イベントで発火）
GAS（Google Apps Script）※プログラムを挟む中継役
  ↓ Lark API（直接呼び出し）
Lark Bitable（ベース）
  ↓ ボタンクリックトリガーの自動化
公式LINE（1対1メッセージ送信）
```

#### 実施内容の詳細

1. **友達情報の自動登録**
   - 公式LINEに友達追加されたタイミングでWebフック（外部イベント発生時に自動通知を送る仕組み）が発火。
   - 発火先をGASのデプロイ済みアプリに設定し、友達情報（LINEユーザーID・プロフィール表示名等）をLark Bitable（ベース）に登録。
   - LarkのAPIを直接呼び出してレコードを登録（自動化機能は使用せず）。

2. **1対1メッセージ送信**
   - ベース上のレコード（友達1件）に対して「メッセージ送信」ボタンをクリックすることで、その友達へ個別メッセージを送信。
   - Lark自動化（ボタンクリックトリガー）を設定して実現。

3. **一斉配信テーブル**
   - 友達全員への一斉配信用テーブルも別途用意（デモ時点では自動化未設定）。

#### GAS（Google Apps Script）を採用した理由

- Webフック受信からベースへのデータ登録処理を記述するためにプログラムが必要。
- 比較的簡単に実装できるためGASを採用。
- 将来的にはGASを使わない構成も可能。

#### Lark自動化を使わずAPIを直接叩いた理由

- Lark自動化の実行回数節約のため。例として、1日10人が友達登録する場合、自動化だと10回消費される。

#### L ステップ・宴（うたげ）等の他ツールとの関係

- LINE公式アカウントのメッセージングAPIを共通の起点とするため、ベースとL ステップ・宴等の外部LINE拡張ツールを**並列連携することは基本的に不可**（各ツールがAPIを非公開にしているため干渉する）。

### 3-3. MCPを活用したLark Bitable作成 ― JSON活用の説明

**28:00頃 ～ 56:00頃**

遠藤氏より、MCPを使ってClaude（AI）にLark Bitableを自動作成させる手法について説明が行われた。

#### JSONデータを使う理由

- **JSON（JavaScript Object Notation）**：データを構造化して表現するフォーマット。
- AIに自然言語（日本語）のみで複雑な指示を出すと出力精度が落ちる。
- JSONで構造化したデータを渡すことでAIがより正確にベースを構築できる。

#### JSON構成（各ステップの役割）

| ステップ | 内容 |
|----------|------|
| Step 1：Create Base | ベース全体の定義・作成 |
| Step 2：Create Table | テーブルの作成 |
| Step 3：Create Field | フィールドの作成（フィールドタイプ番号の指定が必要） |
| Step 4：Create Record | レコード（データ）の登録 |
| Step 5：カスタマイズ | 任意（ビューカスタマイズ等） |

#### フィールドタイプ番号（Type Code）について

LarkのAPIでフィールドを作成する際、フィールドの種類を番号で指定する必要がある。

| 番号 | フィールドタイプ |
|------|----------------|
| 1 | テキスト（Text） |
| 2 | 数値（Number） |
| 3 | 単一選択（Single Select） |
| 4 | 複数選択（Multi Select） |
| 5 | 日付（Date） |
| 11 | （文脈上：メンバー等） |
| 21 | （文脈上：その他） |

> **参照元：** Lark公式デベロッパーサイト（英語）のフィールドタイプ定義ページ。正確な情報はこの公式ページを参照することが推奨された。

#### JSONの変更可能箇所と変更不可箇所

| 区分 | 対象箇所 | 備考 |
|------|---------|------|
| **変更可** | ベース名・テーブル名・フィールド名・選択肢テキスト・レコード内容 | 自由に書き換えてよい |
| **変更不可** | `_tool` （アンダースコアツール）の値 | 認証・認可に使うツール識別子のため固定 |

#### AIへの指示出し方（プロンプト例）

遠藤氏が実際に使用したプロンプト（Cursorでの使用例）：

```
新規のBitableデータベースを作成するリクエストJSONのサンプルを作成してください。
条件として、複数テーブルかつそれぞれのテーブルに50レコード追加するJSONにします。
ベースはプロジェクト管理のベースで、テーブルは以下の3つにします：
- 案件管理テーブル
- プロジェクトメンバーテーブル
- タスク管理テーブル
```

> **ポイント：** Lark公式デベロッパーサイトの**一次情報（フィールドタイプ定義等）をAIに渡す**ことで出力精度が向上する。

#### Claudeのスキルズ（カスタム知識）機能の活用

- Claudeのプロジェクト機能（ChatGPTのGPTsに相当）に、Lark公式ドキュメントの情報をナレッジとして登録しておくことで、毎回の指示が簡略化できる。
- 一度設定すれば、以降のベース作成時に自動的に参照してくれる。
