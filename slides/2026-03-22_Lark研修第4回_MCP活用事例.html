2. 研修の目的と背景

  • ▶ 目的
  • MCP（Model Context Protocol：AIと外部システムをつなぐ標準通信規格）の活用事例を理解する。
  • 前回接続したLarkのMCPを実際に使い、Lark Bitable（以下「ベース」）をAIで作成する体験をする。
  • 受講者からの要望（活用事例・外部連携の具体例を知りたい）に応えるコンテンツを提供する。
  • ▶ 背景
  • 本研修は全4回の講義形式で構成されており、**今回が講義形式の最終回**。
  • 次回は「作業会」を1回挟み、その後「発表会」という流れに変更することが寺山氏と遠藤氏の間で合意されている。
  • 前回（第3回）はMCPの概念説明とClaude（Anthropic社のAI）およびLarkのMCP接続設定を実施済み。

3. 研修内容

  • ▶ 3-1. 前回（第3回）の振り返り ― MCPの概念整理
  • **02:19頃 ～ 08:20頃** 遠藤氏より、MCPの概要について以下のとおり再確認が行われた。 | 項目 | 説明 | |------|------| | **MCP（Model Context Protocol）とは** | LLMアプリ（ChatGPT・Claude・Cursorなど）と外部システムをつなぐための**標準通信規格**。HTTPに類似した通信プロトコル。 | | **MCPがない場合** | 各AIアプリが外部サービス1つひとつに個別接続設定が必要。 | | **MCPがある場合** | MCPサーバーを1つ挟むことで、ChatGPT・Claude・Cursor等がすべて同一設定で接続可能になる。 | | **構成要素（5要素）** | ①リモートサービス、②ローカルデータソース、③MCPサーバー、④MCPクライアント、⑤MCPホスト | > **補足：** 5つの構成要素の詳細は「概念として掴む程度で問題なし」と遠藤氏よりコメントあり。 ---
  • ▶ 3-2. 活用事例紹介 ― 公式LINEとLark Bitableの連携
  • **08:20頃 ～ 23:51頃** 遠藤氏より、実際に構築した公式LINEとLark Bitable（ベース）の連携システムについてデモを交えて説明が行われた。 ``` 公式LINE ↓ Webフック（友達追加イベントで発火） GAS（Google Apps Script）※プログラムを挟む中継役 ↓ Lark API（直接呼び出し） Lark Bitable（ベース） ↓ ボタンクリックトリガーの自動化 公式LINE（1対1メッセージ送信） ``` 1. **友達情報の自動登録**
  • 公式LINEに友達追加されたタイミングでWebフック（外部イベント発生時に自動通知を送る仕組み）が発火。
  • 発火先をGASのデプロイ済みアプリに設定し、友達情報（LINEユーザーID・プロフィール表示名等）をLark Bitable（ベース）に登録。
  • LarkのAPIを直接呼び出してレコードを登録（自動化機能は使用せず）。
  • 2. **1対1メッセージ送信**

4. 質疑応答（主なもの）

  • ▶ Q1. 「友達情報」とは何を指すのか？
  • | 項目 | 内容 | |------|------| | **質問者** | 眞子（Mako）氏 | | **質問内容** | 「ここでの友達情報とは何を意味しますか？」「送られてきたメッセージのテキスト内容のことですか？」「LINEの登録名ということで大丈夫ですか？」 | | **回答（遠藤氏）** | テキスト内容ではなく、LINEユーザーIDとプロフィール表示名（LINE登録名）のことを指す。公式LINEに友達追加した全ユーザーの識別情報（ID・名前）がベースに蓄積される仕組み。なお、公式LINEアカウントごとに異なるIDが振られるため、複数の公式LINEに登録しても識別が可能。 | ---
  • ▶ Q2. GASを経由することの意味は？
  • | 項目 | 内容 | |------|------| | **質問者** | 竹村氏（コメント欄から） | | **質問内容** | 「GASを使用しなくてもLarkに接続できるが、GASを経由することでメッセージ送信等ができるようになるイメージで合っているか？」 | | **回答（遠藤氏）** | ほぼ正しい理解だが補足あり。メッセージ送信自体はGASを使わずに直接公式LINEに送ることも可能。今回GASを使ったのは、Webフックで受け取った友達情報をベースに登録する処理を自動化機能を使わずに実現するためのプログラム記述が必要だったため。また自動化の実行回数を節約する目的もある。 | ---
  • ▶ Q3. L ステップ・宴等と Lark Bitable の並列連携は可能か？
  • | 項目 | 内容 | |------|------| | **質問者** | 寺山 大夢氏 | | **質問内容** | 「LステップやL MEMEなどの拡張ツールを使っているアカウントでも同じ設計が可能か？干渉するか？」 | | **回答（遠藤氏）** | **干渉する。** LステップやうたげはすべてLINEのメッセージングAPIを起点としており、一方の設定をすると他方が使えなくなる。LステップとLark Bitableを連携させることは、LステップがAPIを公開していないため基本的に不可。 | ---
  • ▶ Q4. JSONを使う理由・使わなくていいケースは？
  • | 項目 | 内容 | |------|------| | **質問者** | 寺山 大夢氏・MANABU氏 | | **質問内容** | 「以前クロードに普通にベース作りたいと言ったら自動で作れたが、今回のJSONとはどう違う？」「JSONを直接書けないメンバーはどうすればよいか？」 | | **回答（遠藤氏・寺山氏）** | MCPを使って作ること自体は同じ。JSONは**精度向上のための手段**。実務的には①自然言語でClaudeにJSONを生成させ、②生成されたJSONでベースを作成する流れが現実的。受講者全員がJSONを手書きできる必要はない。 | ---
